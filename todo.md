## Production-readiness: schema-on-read + concept-map architecture

Goal: stop requiring a code edit (`_FIELD_MAP`/`_ROW_FIELD_MAP`) for every new
CSV/Excel column, while keeping fast/reliable lookups for the columns that
matter. Approach agreed: a generic raw-landing table (schema-on-read) + a
data-driven concept-mapping table (replaces hardcoded dicts) + the existing
curated tables (patients/admissions/records), materialized FROM the mapping
instead of from hardcoded extraction logic. Schema seen by the LLM/text-to-SQL
fallback stays constant regardless of how many files are ingested. Roll out
incrementally, each phase additive so nothing existing breaks.

- **Phase A** — Add `raw_rows(id, source_file, row_index, entity_key,
  data_json)` table. Every ingested CSV/Excel row is also written here as JSON
  (in addition to today's pipeline, unchanged). Add `concept_map
  (source_column_name, canonical_concept, value_type)` table and seed it from
  the current `_FIELD_MAP`/`_ROW_FIELD_MAP`/`_ROW_FIELD_MAP` aliases. Nothing
  existing changes behavior yet — purely additive.

- **Phase B** — Switch curated-table population (`patients`/`admissions`/
  `records`) to be driven by `concept_map` lookups against `raw_rows` instead
  of the hardcoded Python dicts. Adding a new column synonym becomes an
  `INSERT INTO concept_map`, not a code change. Unmapped/new columns from a
  freshly-ingested file get surfaced in an ingestion report for human review
  (and an LLM can *suggest* the mapping for that review — not auto-applied
  unsupervised, to avoid silently merging two different concepts for medical
  data).

- **Phase C** — Add SQLite generated/virtual columns + indexes on `raw_rows`
  for canonical concepts that are common to filter/sort on but not promoted to
  a curated table column (e.g. `json_extract(data_json,'$.deathtime')`). Point
  the text-to-SQL fallback (see below) at `raw_rows` + these generated columns
  plus the curated core tables — this is the stable, bounded schema the LLM
  reasons over no matter how many files have been ingested.

- **Phase D (Postgres migration, when full PhysioNet data arrives)** — Port
  the schema/queries from SQLite to Postgres. This is a port, not a rewrite:
  the table design (raw_rows/concept_map/patients/admissions/records) carries
  over directly. What changes: `AUTOINCREMENT` → `SERIAL`/`IDENTITY`, SQLite
  `json_extract()` → Postgres `->`/`->>`/`#>>` JSONB operators (and `data_json`
  becomes a real `JSONB` column with GIN indexes instead of TEXT), and
  `db/schema.py`'s per-call `sqlite3.connect()` → a real connection pool
  (psycopg2/asyncpg pool or SQLAlchemy). The raw_rows+JSONB approach is
  *better* on Postgres than SQLite (GIN indexes on JSONB are more efficient
  than SQLite's JSON1/TEXT), so Phase C's design choice pays off more at this
  scale, not less.

## Text-to-SQL fallback (deferred until Phases A-C land)

- New retrieval path, used ONLY when the existing fixed `sql_retriever.lookup()`
  returns nothing — existing patient/admission/provider lookups are unchanged
  and remain primary.
- SQL generated by the existing local Mistral/Ollama model
  (`rag/llm_client.call_llm`), grounded on the schema from Phases A-C
  (raw_rows + concept_map + curated tables — stays small/stable as more files
  are ingested).
- Production safety: read-only DB connection (`PRAGMA query_only=1` /
  `mode=ro`), validate generated SQL with `sqlglot` (single `SELECT` only,
  whitelist tables), enforce `LIMIT` + timeout, one retry with the SQL error
  fed back to the LLM on failure.
- Observability: wrap execution with `time.perf_counter()`; for SQLite, run
  `EXPLAIN QUERY PLAN` alongside the real query to report which tables/indexes
  were touched and approximate rows scanned. Return as metadata alongside the
  answer (e.g. `query_time_ms`, `tables_touched`).

## Deferred — resume once the full PhysioNet MIMIC-III dataset is available

The demo dataset (`mimic-iii-clinical-database-demo-1.4`, ~100 patients) was
enough to validate the schema/field-mapping work (Phases 1, 3, 4). The large
event tables (CHARTEVENTS, LABEVENTS, INPUTEVENTS_CV/MV, OUTPUTEVENTS,
DATETIMEEVENTS, PRESCRIPTIONS, MICROBIOLOGYEVENTS, CPTEVENTS,
PROCEDUREEVENTS_MV, TRANSFERS — ~900K rows total) were moved out of
`bucket/uploads/` into `bucket/uploads/mimic_large_not_yet_ingested/` because
synchronous per-row ingestion (one SQLite connection per row via `get_db()`)
made `/ingest` take hours and time out the 120s frontend request.

The following phases are intentionally on hold until the real, full PhysioNet
dataset is downloaded and we're ready to ingest it for real:

  text (the demo's NOTEEVENTS.csv is empty, so this couldn't be tested
  meaningfully).
- **Phase 5** — Full Docker-stack end-to-end validation against the complete
  dataset (not just the ~15 small demo CSVs).
- **Phase 6** — Benchmarking, membership-inference / differential-privacy
  evaluation — needs realistic data volume to be meaningful.

### Before re-attempting ingestion of the large event tables, fix:

1. **`db/schema.py` `get_db()` performance** — opens/commits/closes a new
   SQLite connection per row via `insert_record()` /
   `find_or_create_admission()`. At ~900K rows this is the root cause of the
   ingestion timeout. Needs either a persistent/pooled connection, batched
   writes, or a background job (Celery/RQ) with progress reporting instead of
   a synchronous `/ingest` HTTP call.

2. **`admissions` table catch-all** — unlike `records` (which has
   `details_json` for any column not in `_ROW_FIELD_MAP`), `admissions` is a
   rigid fixed-schema table. Any ADMISSIONS-style column not explicitly
   mapped is silently dropped from admission-summary results.
   *Superseded by Phase A/B above* — once `raw_rows` exists, every
   ADMISSIONS-style column is captured automatically regardless of table
   rigidity; no separate `details_json` add needed for `admissions`.

3. **`_extract_specific_field` normalization** — currently does
   `field_lower in col.lower()` substring matching, which fails for
   "death time" vs "deathtime" (space vs no space). Normalize both sides
   (strip spaces/underscores, lowercase) before comparing so new field names
   are found without hand-maintained alias lists.
   *Still worth doing independently* — improves the generic fallback for the
   curated tables even before/regardless of Phases A-C.

4. **Regex intent-parser fallback visibility** — the LLM normally extracts
   `specific_field` from natural language; the regex `field_map` in
   `intent_parser.py` is only a fallback for when the LLM call fails. Add
   logging/metrics for how often the fallback is used in practice — if it's
   common, the hand-maintained keyword list becomes a real maintenance burden
   at full-dataset scale.
